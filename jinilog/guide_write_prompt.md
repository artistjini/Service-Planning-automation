```
프로젝트 루트/
└── jinilog/
    ├── dev_guide.md          ← 개발자 교과서
    ├── user_guide.md         ← 사용자 설명서
    ├── dev_user_blog.md      ← 블로그 통합본
    └── guide_write_prompt.md ← 이 파일 (작성 지침)
```

## 2. dev_guide.md 작성 프롬프트

### 역할 설정
> "너의 역할은 친절하고 상세히 차례차례 가르쳐 주는 코딩 전문가이자 선생님이야."

### 독자 수준
- **초등학생도 이해할 수 있는 수준**
- 어려운 개념이 나오면 바로 옆에 쉬운 비유와 예시를 달 것
- 전문 용어는 처음 등장 시 반드시 한 줄 설명 추가

### 필수 포함 항목 (이 순서로 작성)

#### 항목 1: 프로젝트 목적
- 한 줄 설명 (One-liner)
- 왜 만들었는지 배경
- 어떤 문제를 해결하는지
- 비유로 설명 (게임 미니맵, 칠판 등 초등학생에게 친근한 예시)
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 2: 활용 방법
- 이 프로그램이 하는 일 (데이터 흐름 다이어그램 포함)
- 어떤 상황에서 쓰는지
- 안 하는 일 (NON-GOALS) 표로 정리
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 3: 프로그래밍 코딩 기초 개념 설명
각 개념마다:
- 개념 이름과 한 줄 정의
- 초등학생 비유
- 실제 코드 예시 (짧게)
- 이 프로젝트에서 어떻게 쓰였는지

다뤄야 할 개념 목록 예시  (프로젝트에 따라 추가/제거):
- 언어 특성 (예: TypeScript의 타입)
- 플랫폼/런타임 (예: VS Code Extension API)
- 파일 감시 + 디바운스
- 파서(Parser) + 정규표현식
- Webview + CSP 보안
- 이벤트 버스
- 단방향 데이터 흐름
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 4: 기능 개발 순서와 SOP
- 전체 Phase/단계 표 (이름, 하는 일)
- 각 단계별 표준 절차(SOP):
  - 이 단계에서 만드는 것
  - 작성 순서
  - 실제 이 프로젝트 예시 코드/문서 발췌
- 개발 순서의 이유 설명 ("왜 이 순서인가요?")
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 5: 사용된 원본 소스와 출처 URL
- 라이브러리 표 (이름 / 역할 / 출처 URL)
- 플랫폼/내장 모듈 표
- 디자인 영감 출처
- 개발 방법론 출처
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 6: 전체 알고리즘 로직 (파이 구조)
- 전체 흐름도 (ASCII 다이어그램)
- 파서 알고리즘 상세 흐름
- 화면 렌더링 알고리즘
- 레이어별 파이(Pie) 구조 다이어그램
  ```
  최상위 (사용자)
      ↓
  데이터 원본 (파일)
      ↓
  데이터 처리 레이어
      ↓
  로직 조율 레이어
      ↓
  UI 출력 레이어
  ```
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 7: 개발 히스토리 (시간순)
- 날짜별로 무엇을 했는지
- 각 날짜에 내린 핵심 결정 (ADR 등)
- 버전별 변경 내용
- 형식: `날짜 — 버전 — 한 일 — 핵심 결정`
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 8: 에러와 해결 방법
각 에러마다:
- **에러 제목** (에러 #N)
- 언제 발생했는지
- 증상
- 원인 분석 (코드 예시 포함)
- 해결 코드
- **배운 점** (한 줄)
- **→ 항목 끝에 📌 요약 박스 추가**

#### 항목 9: Key Takeaway · 주의사항 · 개발 팁
- Key Takeaway: 💡 교훈 N개 (번호 붙여서)
- 주의사항: ⚠️ 경고 N개 (코드 블록 형식)
- 개발 팁: 팁 N개 (코드 예시 포함)
- **→ 항목 끝에 📌 요약 박스 추가**

#### 부록: 전체 프로젝트 파일 구조
- ASCII 트리 형식
- 각 파일/폴더 한 줄 설명

### 문서 스타일 규칙
- 모든 코드 블록에 언어 표시 (` ```typescript `, ` ```markdown ` 등)
- 중요 개념은 **굵게**
- 표는 최대한 활용 (항목 비교, 목록 정리에 표 사용)
- 각 항목 앞에 `## N. 제목` 형식 사용
- 파일 맨 끝에 작성 날짜 및 버전 표시

## 3. user_guide.md 작성 프롬프트

### 역할 설정
> "사용자가 초등학생 수준이라고 가정하고 쉽고 친절하게 차례차례 설명해 줘."

### 독자 수준
- 코딩을 모르는 사람
- 컴퓨터를 처음 쓰는 사람도 이해할 수 있는 수준
- 전문 용어 사용 금지 (불가피하면 괄호 안에 쉬운 설명 추가)

### 필수 포함 항목

#### 항목 1: 이 프로그램이 뭔가요?
- 한 마디 설명 (초등학생 언어)
- 비유 표 (게임, 스마트폰, 학교 등 친근한 것과 비교)
- 화면 구성 ASCII 미리보기

#### 항목 2: 설치 전 준비물 확인
- 준비물 표 (준비물 / 확인 방법 / 없으면 어떻게)
- 각 준비물 쉬운 설명 (Q&A 형식)

#### 항목 3: 설치 방법 (단계별)
- Step N 형식으로 번호 붙이기
- 각 Step에 구체적 행동 지시 ("클릭", "입력", "선택")
- 성공 확인 방법 포함
- 여러 방법이 있으면 방법 A / 방법 B로 구분

#### 항목 4: 첫 번째 사용 — 기본 설정
- 파일 구조 이해 표 (파일 / 역할 / 없어도 되나?)
- 핵심 파일 작성 예시 (실제 내용 포함)
- 기호/표시의 의미 표

#### 항목 5: 화면 구성 이해하기
- 각 섹션/탭 ASCII 다이어그램
- 섹션별 설명 표 (보여주는 것 / 업데이트 시점)
- 클릭했을 때 일어나는 일

#### 항목 6: 매일 쓰는 방법
- 시작 루틴 (번호 목록)
- 파일 업데이트 방법 (전/후 코드 비교)
- 각 기능 쓰는 방법 (단계별)

#### 항목 7: 자주 묻는 질문 (FAQ)
- 7~10개 Q&A
- 각 Q에 원인 + 해결 표 또는 단계별 해결법
- `---` 구분선으로 각 Q 분리

#### 항목 8: 유의할 점과 주의사항
- 중요한 규칙 (코드 블록 형식)
- 알아두면 좋은 것들 표 (항목 / 내용)
- 보안 관련
- 도움이 필요할 때 표 (상황 / 해결 방법)

#### 부록: 빠른 참조 카드
- 프린트해서 붙여둘 수 있는 요약 박스
- 자주 쓰는 단축키, 기호 의미, 탭 역할 포함

### 문서 스타일 규칙
- 이모지는 절제해서 사용 (강조 포인트에만)
- 표는 적극 활용 (중간중간 자주 넣기)
- 코드 예시는 최소화, 꼭 필요하면 설명과 함께
- 각 Step은 한 행동씩 (여러 행동 묶지 않기)

## 4. dev_user_blog.md 작성 프롬프트

### 형식
- **블로그 글 형식** (개인 기술 블로그 스타일)
- 제목: 프로젝트를 통해 배운 것 중심으로
- 태그: 관련 기술/개념 키워드

### 필수 포함 항목

#### 들어가며
- **이 글을 쓰게 된 계기** 를 솔직하게
- 프로젝트의 출처/배경 명확히 기재
- 내가 한 역할과 남이 한 역할 구분

#### 프로젝트 한 줄 소개
- One-liner
- 주요 기능 목록 (bullet)

#### 나의 역할 (본인의 기여 범위 명확히)
- 무엇을 받았는지
- 무엇을 직접 했는지
- 무엇을 분석했는지

#### 코드를 분석하며 배운 것들
- **배움 N: 제목** 형식으로 N개 나열
- 각 배움마다:
  - 코드에서 발견한 것
  - 왜 그렇게 됐는지
  - 비유로 설명
  - 코드 예시 (짧게)
  - 실생활 적용 포인트

#### 설치하면서 배운 실용적인 것들
- 설치 과정에서 배운 것 (직접 경험)
- 에러와 해결 (직접 만난 것)

#### 전체 타임라인 정리
- 원작자 작업 구간과 내 작업 구간 명확히 구분

#### 마무리
- 이 프로젝트에서 가장 크게 배운 것 한 문장
- 감사 인사 (출처/공유자)
- 앞으로의 계획

#### 참고 자료 표
- 자료 / 링크 표 형식

#### 문서 끝 메타 정보
```
작성: YYYY-MM-DD
프로젝트 버전: vX.X.X
원작자: (이름 + 출처)
분석 및 가이드 작성: (본인 이름/핸들)
```

### 블로그 스타일 규칙
- 독자에게 말하는 말투 ("~해요", "~예요")
- 개인적인 소감과 발견의 기쁨을 표현
- 기술 개념은 비유로 먼저, 코드는 나중에
- 너무 길어지면 가독성이 떨어지므로 각 섹션 400자 이내 권장
