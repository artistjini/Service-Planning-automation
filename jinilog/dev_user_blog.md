## 이 프로젝트가 뭔가요?

**Blueprint Dashboard**는 AI와 함께 코딩할 때 생기는 불편함을 해결하는 도구예요.

AI(Claude)와 함께 작업하다 보면 이런 일이 자꾸 생겨요:
> *"어제 어디까지 했더라?"*
> *"지금 몇 번째 단계인지..."*
> *"다음에 뭘 해야 하지?"*

이 프로그램은 `.blueprint/state.md`라는 메모 파일을 읽어서, IDE 사이드바에 현재 진행 상황을 항상 보여줘요. 마치 게임의 미니맵처럼요.

**주요 기능:**

| 기능 | 설명 |
|---|---|
| 사이드바 상태 표시 | Phase 진행도, 현재 단계, 다음 할 일 |
| 4탭 가운데 화면 | Plan / Spec / Preview / Errors |
| 파일 자동 감지 | 저장하면 0.2초 내 화면 자동 업데이트 |
| 완전 로컬 | AI 호출 없음, 인터넷 불필요 |

## 코드를 분석하며 배운 것들

### 배움 1: 코딩 전에 문서 세 개를 먼저 쓴다

코드를 열어보기 전에, `docs/` 폴더 안에 이미 세 개의 문서가 완성되어 있었어요.

```
docs/PRODUCT.md      → 무엇을 왜 만드는지
docs/DESIGN.md       → 어떻게 생겼으면 하는지
docs/ARCHITECTURE.md → 코드 구조를 어떻게 짤지
```

처음에는 "문서가 뭐 그리 중요해, 그냥 코딩하면 되지"라고 생각했어요.

그런데 코드를 읽으면서 이 세 문서가 얼마나 중요한지 알게 됐어요. 코드 한 줄 한 줄이 이 문서들의 결정에 근거하고 있었거든요. 설계도 없이 집을 지으면 나중에 다 뜯어고쳐야 하는 것처럼, **코딩도 설계가 먼저**라는 걸 실감했어요.

---

### 배움 2: "안 하는 것"이 "하는 것"만큼 중요하다

PRODUCT.md에 **NON-GOALS** 섹션이 있어요. "이 프로그램이 절대 하지 않을 것들" 목록이에요.

```
NON-GOALS:
- AI 호출 직접 안 함
- 파일 수정 안 함 (읽기만)
- 팀 협업 기능 없음
- 마켓플레이스 배포 안 함
```

처음에는 이게 왜 중요한가 싶었는데, 코드를 보니 이해가 됐어요. 이 목록이 있기 때문에 코드가 단순하고 명확해요. "이건 안 한다"를 정해두지 않으면 기능이 끝없이 늘어나서 결국 아무것도 제대로 못 만들게 돼요.

**한 가지 기능에 집중하는 용기** — 이게 좋은 제품의 비밀이구나 싶었어요.

---

### 배움 3: 5개 부서로 나눈 코드 구조

`src/` 폴더를 열면 이렇게 나뉘어 있어요:

```
src/
├── parser/        → 파일 읽고 데이터로 변환
├── file-watcher/  → 파일 변경 감지
├── sidebar/       → 사이드바 화면
├── webview/       → 가운데 탭 화면
└── extension.ts   → 전체 연결 (지휘자)
```

각 폴더(도메인)가 자기 역할만 해요. `parser`는 파싱만, `sidebar`는 화면만, `extension.ts`가 다 연결해요.

처음에는 "왜 이렇게 나눠놨을까?"했는데, 분석하다 보니 이점이 보였어요. **한 부분을 고쳐도 다른 부분이 안 망가져요.** 학교로 치면, 국어 선생님이 바뀐다고 수학 수업이 영향받지 않는 것처럼요.

이걸 **도메인 분리**라고 하고, 좋은 코드의 기본 원칙 중 하나예요.

---

### 배움 4: 데이터는 한 방향으로만 흐른다

이 프로그램에서 데이터가 흐르는 방향은 항상 같아요:

```
파일(.md) → 파서 → UI 화면
```

화면에서 파일을 수정하는 코드는 어디에도 없어요. "읽기 전용"이에요.

처음에는 당연한 것 같았는데, 이 규칙이 깨지면 어떻게 되는지 상상해보니 이해됐어요. 화면이 파일도 바꾸고, 파일이 화면도 바꾸면 — 어디서 뭐가 바뀐 건지 추적하기 어려워져요. **단방향 흐름**은 버그를 찾기 쉽게 해주는 규칙이에요.

---

### 배움 5: 파서(Parser) — 텍스트를 데이터로 바꾸는 번역가

`src/parser/state.ts`를 읽을 때 가장 흥미로웠어요.

`state.md` 파일에 사람이 이렇게 쓰면:
```
- [x] Phase 0: PRODUCT (2026-05-22)
```

파서가 이걸 읽어서:
```typescript
{ id: 0, name: "PRODUCT", status: "done", completedAt: "2026-05-22" }
```
이런 데이터 객체로 바꿔요.

여기서 **정규표현식(RegEx)**을 처음 제대로 봤어요. 패턴으로 텍스트의 특정 부분을 골라내는 도구인데, 처음엔 암호같아 보였지만 하나씩 뜯어보니 규칙이 있었어요.

또 버전별 호환성 처리 코드도 인상적이었어요. Phase 번호가 버전마다 달라져도 예전 파일을 읽을 수 있게 매핑 테이블을 만들었어요. 실제 서비스에서 "예전 것도 잘 읽힌다"는 게 얼마나 중요한지 배웠어요.

---

### 배움 6: 디바운스 — 쏟아지는 이벤트를 하나로

`file-watcher/watcher.ts`에 이런 코드가 있어요:

```typescript
const DEBOUNCE_MS = 200;

// 파일 저장할 때마다 타이머 리셋
// 200ms 동안 조용해지면 그때 처리
setTimeout(() => handleFileChange(), DEBOUNCE_MS);
```

파일 하나를 저장하면 OS가 변경 이벤트를 수십 번 보내거든요. 이걸 다 처리하면 화면이 계속 깜빡여요.

**디바운스**가 이걸 해결해요. 0.2초 동안 기다렸다가 마지막 것만 처리해요. 버튼을 여러 번 눌러도 0.2초 기다렸다가 한 번만 울리는 초인종처럼요.

나중에 찾아보니 이건 웹 개발에서 검색창 자동완성, 스크롤 이벤트 처리 등 아주 많은 곳에서 쓰이는 패턴이었어요.

---

### 배움 7: CSP — Webview 보안 정책

코드에 이런 줄이 있어요:

```typescript
const csp = `style-src ${webview.cspSource} 'unsafe-inline';`;
```

처음엔 뭔지 몰랐는데, DIGEST.md를 읽으니 이런 에러 기록이 있었어요:

> "sidebar progress bar CSP 버그 수정 (v0.9.3) — `unsafe-inline` 누락이 root cause"

`unsafe-inline`이 없으면 인라인 스타일(`style="width: 80%"`)이 보안 정책에 막혀서 progress bar가 0%로 보이는 버그였어요. VS Code Webview는 기본적으로 보안이 매우 엄격하다는 걸 이렇게 배웠어요.

**버그를 수정한 코드보다, 왜 그 버그가 생겼는지가 더 중요한 공부** 가 됐어요.

---

### 배움 8: ADR — 왜 이렇게 만들었는지 기록하기

`docs/adr/` 폴더에 10개의 ADR(Architecture Decision Record) 파일이 있어요.

ADR이 뭔지 처음 알았어요. 코딩 결정을 내릴 때 **"왜 이 결정을 했는지"** 를 기록해두는 짧은 문서예요.

예를 들어:
- **ADR-002**: AI 호출 금지 → 이유: 시각화·동기화만 하는 도구이므로
- **ADR-007**: TreeView → WebviewView 전환 → 이유: 디자인 자유도를 높이기 위해
- **ADR-010**: REVIEW를 Phase로 추가 → 이유: SHIP 전 품질 게이트가 필요해서

이런 결정들이 코드 안에 흔적도 없이 박혀 있는 거예요. ADR이 없었다면 나중에 "왜 이렇게 됐지?"를 절대 알 수 없었을 거예요.

**코드는 "무엇을"만 말해주고, ADR은 "왜"를 말해줘요.** 이 차이가 크다는 걸 배웠어요.

---

### 배움 9: 디자인도 버전이 있다

DIGEST.md를 읽으면 디자인이 어떻게 바뀌어 왔는지 알 수 있어요.

```
v0.9.0~0.9.2: 글래스모피즘 (반투명 유리 효과)
v0.9.4: iOS Settings 플랫 디자인으로 전면 교체
```

"왜 바꿨을까?" 궁금해서 DESIGN.md를 읽었더니 이런 철학이 적혀 있었어요:

> **"색은 데이터에만. 장식 색 0."**
> **"한 번에 한 가지만 강조."**

화려한 것보다 **정보가 잘 보이는 것**이 더 좋은 UI라는 철학이 코드에 그대로 반영돼 있었어요.

**색상 팔레트 (iOS 시스템 컬러에서 따옴):**

| 색 | 코드 | 용도 |
|---|---|---|
| iOS Blue | `#007aff` | 강조, 진행 중 |
| iOS Green | `#34c759` | 완료 |
| iOS Red | `#ff3b30` | 경고, 트리거 |
| Page BG | `#f2f2f7` | 배경 |
| Card BG | `#ffffff` | 카드 |

### 배움 10: DIGEST.md — 세션 간 기억을 이어주는 일지

DIGEST.md는 각 작업 세션이 끝날 때 자동으로 기록되는 일지예요.

```markdown
## 2026-05-30 14:51
- 플랫 iOS Settings 시안 전면 적용 (v0.9.4)
- 핵심 결정: accent #007aff 확정
- 다음에 이어갈 것: 사이드바 정보 위계 정리
```

이 파일이 없었다면, 왜 글래스모피즘에서 iOS 스타일로 바꿨는지, 어떤 결정 과정이 있었는지 전혀 알 수 없었을 거예요.

"**다음에 이어갈 것**"을 항상 적어두는 습관 — 다음 날 어디서 시작할지 고민하는 시간을 없애주는 작은 규칙이에요. 이런 습관도 배울 점이었어요.

## 설치하면서 배운 실용적인 것들

### .vsix 설치 경험

처음 `.vsix` 파일을 설치할 때 "이게 뭔가?" 했는데, 스마트폰 앱 설치 파일(`.apk`)과 같은 개념이라는 걸 이해하고 나니 쉬웠어요.

Antigravity IDE에서:
1. Extensions 패널 → `...` → `Install from VSIX`
2. 파일 선택 → 설치
3. 사이드바에 나침반 아이콘이 생기면 성공

### .vscodeignore — 배포 파일 정리의 중요성

6월 1일에 `.vsodeignore`에 `.claude/**`를 추가했어요.

이게 없으면 `.vsix` 패키지 안에 `.claude/settings.local.json` 같은 개인 설정 파일이 포함될 수 있어요. API 키나 개인 정보가 담긴 파일이 배포 파일에 들어가면 안 되죠.

**배포 전에 항상 "어떤 파일이 포함되나?"를 확인해야 한다**는 실용적인 교훈이었어요.

### GitHub Release — 버전 관리의 시작

v0.9.5를 GitHub Release로 올리면서 버전 관리의 흐름을 직접 경험했어요.

```
코드 변경 → 빌드(npm run build) → 패키징(.vsix) → GitHub Release
```

이 한 사이클을 손으로 해보는 것만으로도 소프트웨어 배포가 어떻게 이루어지는지 감이 잡혔어요.

## 이 프로젝트의 성능 목표 (코드에 명시된 것들)

분석하다 보니 ARCHITECTURE.md에 성능 목표가 표로 적혀 있었어요:

| 동작 | 목표 | 달성 |
|---|---|---|
| 파일 변경 → 사이드바 업데이트 | < 50ms | ~10ms |
| 파일 변경 → 웹뷰 리로드 | < 200ms | ~80ms |
| 확장 활성화 시간 | < 500ms | ~250ms |
| 메모리 사용 | < 80MB | ~35MB |
| 키 입력 지연 | 0ms | 0ms |

"이 프로그램 때문에 타이핑이 느려지면 안 된다"는 사용자 관점의 걱정을 코드를 짜기 전에 이미 설계 문서에 적어둔 거예요. 좋은 설계는 이런 것까지 고려한다는 걸 배웠어요.

## 마무리 — "남의 코드 읽기"의 가치

이 프로젝트를 분석하면서 가장 크게 느낀 건, **좋은 코드는 읽는 것만으로도 공부가 된다**는 거예요.

Chris 님이 이 코드에 담아두신 것들:
- 코드를 짜기 전에 문서 먼저 쓰는 습관
- NON-GOALS로 범위를 지키는 방법
- 도메인 분리로 유지보수를 쉽게 하는 구조
- 단방향 데이터 흐름으로 버그를 줄이는 원칙
- ADR로 결정의 이유를 남기는 문화
- DIGEST로 세션 간 기억을 이어주는 루틴

이 모든 게 한 프로젝트 안에 담겨 있었고, 코드를 읽는 것만으로 그 배움이 고스란히 전해졌어요.

좋은 코드를 공유해 주신 Chris 님, 이 수업을 열어주신 문토에 감사드려요.

**"코딩은 혼자 발명하는 게 아니라, 좋은 코드를 읽고 배우는 것"** 이라는 걸 이번 프로젝트를 통해 배웠습니다.

## 배움 11: 단방향 원칙 — 왜 생겼고 왜 지켜야 하나

분석을 마친 후 문득 궁���해졌어요.

> "이 프로그램은 파일을 읽기만 하는데, 왜 그 규칙이 생긴 걸까?"

ARCHITECTURE.md와 ADR-003을 다시 읽으며 이유를 정리했어요.

**단방향 원칙이 생긴 4가지 이유:**

| 이유 | 설명 |
|---|---|
| 버그 추적 단순�� | 화면이 이상하면 파일만 보면 됨. 양방향이면 "파일이 바꿨나? UI가 바꿨나?" 추적 필요 |
| 신뢰성 | 도구가 사용자 모르게 파일을 바꾸면 안 됨 |
| 정체성 보호 | "보여주는 도구"이지 "편집 도구"가 아님 |
| 교체 용이성 | parser만 바꿔도 sidebar·webview는 영향 없음 |

한 줄로 요약하면:
> **"��일이 진실 원본이고, 화면은 그 파일을 읽어서 보여주는 거울이다. 거울이 원본을 바꾸면 안 된다."**

이 원칙 덕분에 코드가 단순하고 예측 가능하다는 걸 코드를 읽으면서 직접 느꼈어요.

## 배움 12: 원칙을 깰 때는 ADR로 근거를 남긴다

분석 중에 한 가지 아이디어가 생겼어요.

> "Phase 완료할 때마다 state.md를 수동으로 고치는 게 번거롭다. 사이드바에서 바로 체크할 수 없을까?"

그런데 이건 단방향 원칙(ADR-003)에 정면으로 부딪히는 아이디어예요.

그래서 **ADR-011**을 직접 작성해봤어요. 원칙을 무조건 깨는 게 아니라 — **"어떤 조건에서, 어떤 파일만, 어떻게"** 를 먼저 설계하는 과정이었어요.

**ADR-011의 핵심 결정 — 조건부 쓰기(Controlled Write):**

| 항목 | 내용 |
|---|---|
| 쓰기 허용 | `state.md` Phase 상태·Next action, `DIGEST.md` 새 항목만 |
| 쓰기 금�� | PRODUCT·DESIGN·ARCH·코드·ADR 파일 전부 |
| 기본값 | `blueprint.autoWrite: false` (opt-in — 사용자가 직접 켜야 활���화) |
| 안전장치 | 사용자 확인 프롬프트 → atomic write → undo 지원 → 변경 로그 |

여기서 **opt-in**이라는 개념도 배웠어요.

```
opt-in  → 기본 꺼짐. 직접 켜야 작동.  ← ADR-011 채택
opt-out → 기본 켜짐. 싫으면 꺼야 함.
```

스마트폰 앱이 처음 설치될 ��� "위치 정보 허용하시겠습니까?" 팝���이 뜨는 것처럼 — 내 파일을 건드리는 기능은 사용자가 명시적으로 허용해야 한다는 원칙이에요.

**가장 크게 배운 것:** 원칙�� 바꿔야 할 ��는 코드부터 짜는 게 아니라, 왜 바꾸는지·어떤 범위까지만 바꾸는지를 먼저 글로 써야 한다는 것.
